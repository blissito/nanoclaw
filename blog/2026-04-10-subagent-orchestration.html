Sub-Agent Orchestration Is Broken Everywhere

April 10, 2026 · NanoClaw Blog

We wanted parallel sub-agents in NanoClaw: spawn multiple Claude instances inside a container, let them work concurrently, aggregate results. The idea is simple. The reality is that nobody has made this work reliably — not us, not upstream NanoClaw, not OpenClaw.

Here's what we found after researching all three projects.

The Core Problem

When a parent agent finishes its turn (the SDK's query() call completes), all in-process sub-agents are killed silently. No output. No error. They just disappear.

This is not a NanoClaw bug. It's a fundamental issue in how the Claude Agent SDK manages process lifecycles. The parent doesn't wait for children to finish — it can't, because the SDK has no concept of "I'm done talking but my children are still working."

NanoClaw Upstream

Issue #684 ("Agent Swarms: in-process subagents silently terminate when main agent completes turn") has been open since March 4, 2026, labeled Critical, with zero comments. The community has filed related PRs:

  • #886 — A news briefing skill using agent swarms (open, untested)
  • #1295 — A2A (agent-to-agent) protocol support
  • #1202 — Agent trace observability with web UI

There are a handful of GitHub Discussions (#112, #330, #1039) where people ask "has anyone gotten swarms to work?" The answer is consistently no.

OpenClaw: More Plumbing, Same Result

OpenClaw has significantly more sub-agent infrastructure than NanoClaw:

  • sessions_spawn with fire-and-forget and persistent modes
  • sessions_send and sessions_history for inter-agent communication
  • maxSpawnDepth config for nested hierarchies
  • maxChildrenPerAgent limits

But they hit the same wall. Issue #46768: after sessions_spawn, the parent orchestrator goes dormant. Completion events from children queue up but nobody processes them. Users have to manually ping the parent to wake it up.

It gets worse:

  • #52174 — In multi-depth orchestration, aggregated output from depth-1 sub-agents never reaches the main session
  • #43113 — Sub-agents with large context (600K+ tokens) die with "Network connection lost," all work lost
  • #48984 — /subagents kill terminates the caller instead of the target

OpenClaw's proposed fix is "completion-triggered session wake" — the parent should automatically resume when a child finishes. It hasn't been implemented.

What Actually Works Today

The only reliable pattern we've found is scheduled tasks as async sub-agents. Instead of spawning children inside a container:

  1. The agent creates a scheduled task (fires once, immediately or after a delay)
  2. The task runs in its own container with its own lifecycle
  3. Results are persisted to group memory or the database
  4. The original agent (or a future invocation) reads the results

It's not elegant. There's no real-time aggregation. But it works because there's no parent-child lifecycle dependency. Each task is independent.

When Will This Get Fixed?

The fix needs to happen at the SDK level. Either:

  • The Agent SDK needs a "wait for children" primitive that keeps the parent alive
  • Or sub-agents need to be fully decoupled processes with a message bus (which is essentially what our scheduled-task workaround already does)

We're watching NanoClaw #684 and OpenClaw #46768. Until one of them ships a real fix, scheduled tasks are the answer.
